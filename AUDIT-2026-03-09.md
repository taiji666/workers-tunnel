# Workers VLESS Audit

Date: 2026-03-09

This audit reviews the repository implementation only. It does not verify the live deployed Worker endpoint.

## Verdict

- VLESS protocol: partially compliant.
- Cloudflare Workers platform usage: mostly compliant, with one configuration risk and one local-tooling validation gap.
- Overall implementation status: this repository implements a basic Cloudflare Workers VLESS-over-WebSocket TCP proxy, but not a complete VLESS proxy.

## Findings

### High

1. UUID parsing is not strict and silently accepts malformed values.

- Code: [src/lib.rs](/d:/Code/workers-tunnel/src/lib.rs#L136) and [src/lib.rs](/d:/Code/workers-tunnel/src/lib.rs#L165)
- `parse_user_id()` strips every non-hex character instead of validating canonical UUID format and exact 16-byte output.
- A misconfigured `USER_ID` containing extra junk characters can still decode to the same 16 bytes, while a short or malformed value is only discovered later by a failed byte comparison.
- This is not a proven auth bypass in the current code path, but it is a protocol-compliance and safety defect because the server does not strictly validate the configured UUID.

### Medium

2. The project is only a partial VLESS implementation, not a full VLESS proxy.

- Code: [src/lib.rs](/d:/Code/workers-tunnel/src/lib.rs#L210) and [src/lib.rs](/d:/Code/workers-tunnel/src/lib.rs#L315)
- TCP is implemented.
- UDP is only implemented for port `53`, and even then by translating packets to DoH against `https://1.1.1.1/dns-query`.
- Unsupported commands are rejected, and Mux is not implemented.
- Conclusion: the project should be described as a basic VLESS-over-WS TCP proxy with limited DNS-over-HTTPS UDP handling, not a full VLESS server.

3. `compatibility_date` is behind current Workers best practice.

- Config: [wrangler.toml](/d:/Code/workers-tunnel/wrangler.toml#L3)
- The repo uses `compatibility_date = "2024-09-02"`.
- Cloudflare's current guidance says new projects should set `compatibility_date` to the current date and update it occasionally after testing.
- This is not automatically broken, but it is a platform-maintenance risk and means the project is not aligned with the latest recommended Worker runtime baseline.

### Low

4. Repository claims and implementation differ on UDP support.

- Docs: [README.md](/d:/Code/workers-tunnel/README.md#L15)
- Code: [src/lib.rs](/d:/Code/workers-tunnel/src/lib.rs#L315)
- README says UDP proxy is not supported, but the code does implement limited UDP handling for DNS on port `53`.
- This is a documentation defect, not a runtime defect.

5. There are no automated tests covering protocol parsing or Worker behavior.

- Evidence: `cargo test` reports `0 passed; 0 failed`.
- There are no unit or integration tests for valid/invalid VLESS frames, WebSocket upgrade handling, TCP connect failure paths, or DNS UDP behavior.
- This weakens any claim that the repository has been successfully validated beyond compilation.

6. Local deploy validation is incomplete on this workstation.

- Command result: `npm run deploy -- --dry-run` failed locally because `@cloudflare/workerd-windows-64` is missing from the local install.
- This does not prove the repo is broken, especially because CI already runs `npx wrangler deploy --dry-run` in [deploy.yml](/d:/Code/workers-tunnel/.github/workflows/deploy.yml#L44).
- It does mean this local environment did not complete end-to-end deploy validation.

## Protocol Assessment

### What matches VLESS

- Request parsing order in [src/lib.rs](/d:/Code/workers-tunnel/src/lib.rs#L160) matches the documented VLESS request structure:
  - version
  - UUID
  - addons length
  - addons
  - command
  - port
  - address type
  - address
- Address handling supports domain, IPv4, and IPv6.
- For version `0` and empty response addons, the fixed response header `[0x00, 0x00]` in [src/lib.rs](/d:/Code/workers-tunnel/src/lib.rs#L104) is consistent with the common VLESS response shape.

### What does not qualify as full compliance

- UUID validation is not strict.
- UDP support is partial and product-specific.
- Mux is not implemented.
- There is no protocol test suite proving interop with multiple VLESS clients.

## Workers Assessment

### What matches Workers

- The Worker uses `WebSocketPair`, `server.accept()`, and `Response::from_websocket(client)` in [src/lib.rs](/d:/Code/workers-tunnel/src/lib.rs#L76), which matches the standard Workers WebSocket handling model.
- TCP outbound is created inside the fetch handler flow through `Socket::builder().connect(...)` in [src/lib.rs](/d:/Code/workers-tunnel/src/lib.rs#L275), which is the correct lifecycle pattern for Worker sockets.

### Constraints that still apply

- Workers outbound TCP sockets cannot connect to Cloudflare IP ranges.
- Workers cannot open outbound connections to port `25`.
- Workers will reject a TCP loop where the Worker connects back to itself.
- Those platform limits are external constraints on this design, not repository bugs, but they are relevant to whether the proxy is usable for arbitrary targets.

## Build And Validation Evidence

- `cargo check`: passed.
- `cargo test`: passed, but with zero tests.
- `npm run deploy -- --dry-run`: failed locally because the local `workerd` platform package is missing.
- CI workflow includes a dry-run deploy build step on Ubuntu in [deploy.yml](/d:/Code/workers-tunnel/.github/workflows/deploy.yml#L27).

## Final Conclusion

This repository does implement the core of a Cloudflare Workers VLESS-over-WebSocket TCP proxy. It upgrades WebSocket requests, parses VLESS request headers, opens outbound TCP sockets, and bridges traffic bidirectionally.

However, it is not accurate to conclude that the project fully implements a complete VLESS proxy, and it is not possible from this repository alone to conclude that the deployed Worker has been successfully validated in production. The strongest defensible statement is:

> The repository implements a basic Workers VLESS-over-WebSocket TCP proxy with limited DNS UDP support, but only partially satisfies full VLESS-server expectations and lacks strict UUID validation, automated protocol tests, and a successful local dry-run deployment on this workstation.
